阶段 6.2 · 容器与资源调度

前面我们讲了驱动和加速库——它们让单台机器上的 GPU 能高效运转。但智算中心有几百上千台服务器、几千上万张 GPU，同时有几十上百个算法工程师在上面跑任务。这么多人和机器的"撮合"问题，就是容器和资源调度要解决的。

场景：GPU 怎么分给 100 个工程师？

一个智算中心有 1000 台 GPU 服务器、8000 张 GPU。100 个算法工程师同时在做实验——有人要 8 张卡训练大模型，有人要 2 张卡做微调，有人要 1 张卡跑推理测试。怎么公平、高效地把这些 GPU 分配出去？谁来排队？谁优先？

如果每次跑任务都要手动指定用哪台机器、哪张卡，那运维人员会疯掉。你想象一下：每个人来找你——"帮我看看 47 号机现在有没有空卡？""我的任务跑了一半断了，你能帮我重新分配一下吗？"——这根本不现实。

我们需要一个自动化的资源调度系统。它的核心任务就一句话：把用户提交的计算任务，自动分配到最合适的机器上运行。

但在讲调度之前，必须先讲容器——因为容器决定了"任务"长什么样、怎么打包、怎么搬运。

核心概念

Docker / Containerd：把程序"装进箱子"

容器（Container）是什么？最直白的理解：把程序运行所需的一切——代码、库、配置文件、依赖项——打包成一个标准化的"集装箱"。这个箱子拿到任何一台机器上都能直接运行，不需要重新安装任何东西。

为什么 AI 领域特别需要容器？因为机器学习框架（机器学习框架，指 PyTorch、TensorFlow 等用来开发和训练模型的软件工具）的依赖链极复杂。举个例子：你要跑一个 PyTorch 训练脚本，你需要：

  • 特定版本的 Python（比如 3.10，不能用 3.11）
  • 特定版本的 PyTorch（比如 2.1.0）
  • 特定版本的 CUDA（NVIDIA GPU 的并行计算平台，见阶段 6.1）
  • 特定版本的 cuDNN（NVIDIA 的深度学习加速库）
  • 各种第三方 Python 包（numpy、transformers、datasets…）

这些组件版本之间如果有一个对不上，程序就可能报错甚至崩溃。算法工程师的经典噩梦：在自己的笔记本上跑得好好的，推到服务器上就挂了——即传说中的"在我机器上没问题"（"works on my machine"问题）。容器把这个月彻底解决了。

生活类比：
没有容器的时候，就像搬家——把家具、电器、衣服一件件装车，到新地方再一件件拿出来摆好，过程中总有磕碰和丢失。
有了容器，就像用标准集装箱搬家——所有东西提前装进箱子，叉车一叉就走，到地方直接放下就能用。不管目的地是码头还是仓库，箱子的内容物不受影响。

容器 vs 虚拟机：不是一个东西

经常有人把容器和虚拟机（Virtual Machine，简称 VM）搞混。核心区别在于"隔离的层级"：

对比维度容器（Container）虚拟机（VM）
隔离层级进程级——共享宿主机操作系统内核硬件级——每个 VM 有自己的完整操作系统
启动速度秒级（本质上是启动一个进程）分钟级（需要启动完整操作系统）
资源开销很小——只比直接跑进程多一点点大——每台 VM 要占几 GB 内存跑操作系统
镜像大小几百 MB（仅含应用和依赖）几 GB 到几十 GB（含完整操作系统）
GPU 访问直接透传——性能几乎无损需要额外配置 GPU 直通，复杂度更高

关键结论：在 AI 场景下，容器是唯一合理的选择。虚拟机太重了——每台 VM 要多占几 GB 内存去跑一个完整操作系统，对 GPU 内存本就紧张的训练任务来说完全不可接受。而且容器启动快、迁移快，适合 AI 任务"来了就开始算、算完就释放"的特点。

Docker 和 Containerd 的关系

Docker 是这个领域最知名的品牌——它让"容器"这个概念从专业运维圈走进了每一个开发者的日常。Docker 实际上是一整套工具：不仅做容器打包和运行，还管镜像构建、镜像仓库、网络配置等。

Containerd（容器运行时，发音 "container-dee"）是 Docker 把最核心的"运行容器"这部分功能拆出来、捐给开源社区的项目。今天 Kubernetes 默认用 Containerd 而不是 Docker 的完整版——因为编排平台只需要"跑容器"这个功能，不需要镜像构建那些额外功能。

简单记：Docker 是面向开发者的全功能工具，Containerd 是面向生产环境的最小核心。两者底层都依赖同一个行业标准——OCI（Open Container Initiative，开放容器标准），保证各家容器之间可以互通。

Kubernetes（K8s）：容器的"指挥官"

有了容器之后，下一个问题：我有几百上千台服务器，每个容器该放到哪台机器上跑？单靠人手动分配不现实。Kubernetes（简称 K8s，因为 k 和 s 中间有 8 个字母）就是解决这个问题的——它是容器编排平台（容器编排：自动调度、部署和管理大量容器的系统），目前是业界事实标准。

K8s 名字由来：Kubernetes 源于希腊语，意为"舵手"或"领航员"。Google 于 2015 年开源，基于其内部运行了十多年的 Borg 系统。名字很长，业界用 K8s 这个缩写——把首字母 K 和尾字母 s 中间的 8 个字母用数字 8 替换。读法：直接读"K eight s"（K 八 s）。

K8s 帮我们解决什么问题？

  • 自动调度：你提交一个需要 8 张 GPU 的训练任务，K8s 自动找有空闲 GPU 的机器，把容器部署上去。你不需要知道它在哪台机器上跑。
  • 健康监控：容器挂了，自动重启。机器宕机了，自动把任务迁移到另一台机器上。
  • 弹性伸缩：推理服务访问量大增，自动增加副本。夜深人静没人用了，自动缩回去省资源。
  • 服务发现：容器 IP 会变，但 K8s 给你的"服务地址"不变。一个任务的输出可以直接通过固定地址被另一个任务发现和使用。

NVIDIA Device Plugin：让 K8s 知道"我有 GPU"

K8s 天生知道怎么管理 CPU 和内存，但不知道什么是 GPU。需要装一个插件来告诉它。

NVIDIA Device Plugin（NVIDIA 设备插件）是一个 K8s 组件，安装在每台有 GPU 的服务器上。它的工作很简单：扫一遍这台机器上有几张 GPU、什么型号、剩余多少显存，然后把信息上报给 K8s 调度器。这样调度器就知道"这台机器有 8 张 H100，空了 3 张"，做调度决策时就有数据依据了。

如果不装这个插件，K8s 连你机器上有 GPU 都不知道，更谈不上分配。

GPU 虚拟化：把一张大卡"切"成多张小卡

有些任务不需要一整张 GPU。比如一个推理测试可能 2GB 显存就够跑，给一张 80GB 的 H100 太浪费了。这时候需要"切 GPU"。

MIG（多实例 GPU）

Multi-Instance GPU 的缩写。NVIDIA 从 A100 开始支持的技术。在硬件层面把一张物理 GPU 切成多个独立的小 GPU——每个小 GPU 有自己专属的显存、缓存和计算单元，彼此完全隔离。一张 H100 最多可以切成 7 个独立实例。切出来的每个实例看起来就像一张完整的独立 GPU，可靠性高。

vGPU（虚拟 GPU）

Virtual GPU 的缩写。软件层面的 GPU 切分，主要用于较老的 GPU（如 V100 之前的产品）。原理是时间分片——多个任务轮流使用 GPU，每个任务分到一段时间窗口。优点是灵活，缺点是隔离性差：一个任务卡了可能影响其他任务，且性能有波动。

MPS（多进程服务）

Multi-Process Service 的缩写。NVIDIA 提供的一个软件方案，让同一张 GPU 上的多个 CUDA 进程可以"同时"跑，而不是排队。不切硬件，只是让多个进程可以并发访问 GPU。适用于多个小推理任务共享同一张卡的场景。

注意：MIG 是硬件级切分，最可靠但灵活性差（切好的大小是固定的）。vGPU 和 MPS 是软件级方案，灵活但隔离性差。实际选哪个取决于场景：训练任务通常要独占整卡，推理任务适合用 MIG 或 MPS 共享一张卡。

Slurm：传统 HPC 世界的"排队系统"

在 K8s 兴起之前，高性能计算（HPC，High-Performance Computing，即用大量计算机同时算一个任务）领域早就有自己的资源管理工具——Slurm（Simple Linux Utility for Resource Management，发音同 "slurm" 一词，意为"咣当一声"）。

Slurm 的核心模型是作业队列：用户提交一个作业——写明"我要多少台机器、多少张 GPU、跑多久"——Slurm 把它放进队列。当前面的作业跑完了，资源空出来，Slurm 就自动启动下一个等待的作业。

这和 K8s 的哲学有根本区别：

对比维度KubernetesSlurm
核心模型服务型——微服务长期运行，有请求就处理作业型——排队、运行、结束、释放资源
典型场景在线推理服务（API 请求进来—推理—返回结果，7x24 运行）离线训练任务（跑几小时到几天，跑完结束）
任务生命周期长期运行，可能几周几个月不停有明确开始和结束，跑完就停了
资源分配声明式——"我需要 8 张 GPU"，K8s 自动找调度式——进队列排队，轮到你了分配资源
用户群体云原生、互联网公司、AI 平台团队学术界、超算中心、传统科研计算

实际上，很多 AI 集群两者都装：Slurm 管理训练作业的排队和调度，K8s 管理推理服务的长期运行。两者不互斥，各管一块。

技术要点

调度策略：谁先跑？谁排队？

8000 张 GPU，100 个工程师，600 个任务同时排队。谁先跑？怎么排？调度策略决定了集群的资源使用权分配方式。

FIFO（先进先出）

First In First Out。最简单的策略——谁先提交谁先跑。公平但不高效：一个要跑 3 天的大任务排在前面，后面一百个 5 分钟的小任务都得干等。

公平分享（Fair-Share）

按团队或个人历史用量来分配优先级。一个组最近用了很多资源，优先级就降低；另一个组没怎么用，优先级就提高。保证长期来看每个组都能拿到自己"该得"的那份。

优先级队列（Priority Queue）

管理员给不同用户或项目设置不同的优先级权重。VIP 用户的任务优先执行，普通用户等一等。适合企业多部门共享集群的场景。

抢占（Preemption）

高优先级任务来了但资源不够？把正在跑的低优先级任务"暂停"，资源让出来给高优先级的先跑。被暂停的任务保存好当前状态，等资源空出来再恢复——这就是后面要讲的"断点续训"。

Gang Scheduling：要么全部分配，要么一个都不要

这是一个 AI 训练特有的调度需求。Gang Scheduling（群调度，发音 "gang" 即英文"一帮、一群"）的核心意思是：一个分布式训练任务需要 8 张 GPU，这 8 张 GPU 必须同时到位、同时启动。不能给 5 张先跑着，等另外 3 张空出来再加——分布式训练的 GPU 之间每轮迭代都要通信，少一张整个训练停摆。

这和微服务应用完全不同。微服务里先启动 3 个副本，后面再加 2 个完全没问题。训练任务不行——缺一张 GPU 就是缺一张，没法"先跑一部分"。

Gang Scheduling 的实现：

  • Slurm 原生支持：Slurm 天然就是"全部分配一起给"的模型，Gang Scheduling 对它来说是默认行为。
  • K8s 需要扩展：原生 K8s 调度器不支持 Gang Scheduling。需要装额外的调度插件，比如 Volcano（华为开源的 K8s 批调度项目）或 KubeFlow Training Operator。
为什么叫 Gang？Gang 在英文里是"一帮人、一伙人"的意思。Gang Scheduling 的意思是：调度器把这 8 个进程当作一个"帮派"处理——要么 8 个一起上，要么一起等。不会让"帮派"成员分开行动。这比直译的"群调度"更好记——就是"一伙人一起上"。

断点续训：任务被中断了怎么办？

在大规模集群中，任务被中断是家常便饭：机器故障、网络抖动、高优先级任务抢占、系统维护……如果一个训练任务跑了 3 天、还差 2 小时就完了，突然被停掉要从头再来，谁都会崩溃。

Checkpoint（检查点/快照）：训练过程中定期把模型参数（模型学到的所有"知识"）和优化器状态（优化器：训练中自动调整模型参数的工具）保存到存储系统。间隔可以设——每 1000 步存一次，或每 30 分钟存一次。

当任务被中断后，从最近的 Checkpoint 恢复继续训练（即 Resume），相当于从"上次保存的存档"接着打，而不是从第一关重新开始。

关键细节：

  • Checkpoint 要快：大模型的 Checkpoint 可能几百 GB。如果存一次要 20 分钟，每隔 30 分钟存一次，那训练时间等效浪费了近一半。所以需要用快速存储（见阶段 4）或异步保存技术。
  • 指标要一致：8 张 GPU 分布式训练，做 Checkpoint 时必须确保所有 GPU 上的模型参数是一致的。不能存到一半 GPU 还没来得及同步。

GPU 配额管理：公司和团队怎么"分蛋糕"

一个大智算中心通常是多个团队共享的。销售团队不会管技术细节，但他们需要能跟客户讲清楚：资源怎么分、怎么收费、怎么保证"按额分配"。

Resource Quota（资源配额，即每个用户或团队最多能用多少资源的硬限制）是核心机制：

  • 团队 A 拥有 30% 集群配额（即这个团队最多同时用 2400 张 GPU）
  • 团队 B 拥有 20% 配额（即 1600 张）
  • 预留 10% 给紧急任务和系统维护
  • 剩余 40% 作为弹性共享池——先到先得，但不超过各自配额

K8s 用 ResourceQuota 和 LimitRange 两个内置对象实现配额和限额管理。Slurm 用 QOS（Quality of Service，服务质量等级，定义每个用户的资源上限和优先级）和 Account（账户，按部门或项目归集资源用量和计费）体系做更细粒度的按小时/按 GPU 数计费。

配额 vs 限额：配额是"团队总共能用多少"（上限），限额是"单个任务最多能用多少"（单任务上限）。两者搭配使用：团队配额防止某个团队占光所有资源；单任务限额防止某个人的一个任务把整个团队配额都吃光——其他队友都没法干活了。

趋势观察 — NVIDIA 收购 Run:ai 意味着什么？
NVIDIA 2024 年收购 Run:ai 是 GPU 资源调度领域的一个标志性事件。NVIDIA 之前卖 GPU 硬件和网络设备，软件层（怎么调度和分配 GPU）是留给客户和第三方去解决的。收购 Run:ai 意味着 NVIDIA 要把调度层也纳入自身体系——从 GPU 硬件、NVLink 网络、CUDA 软件，一路延伸到"谁来用、怎么用"的调度层。全栈控制又加深了一层。对客户来说，一站式方案更方便；对竞争对手来说，进入壁垒又高了。

阶段 6.2 总结

容器让算法工程师不用关心底层机器——代码打包成"集装箱"，到任何机器上都能直接跑。调度系统让成百上千人公平、高效地共享 GPU 资源——谁先跑、谁排队、谁优先、谁暂停，全由系统自动管理。

  • 容器（Docker / Containerd）解决了"环境一致性问题"——告别"在我机器上能跑"的噩梦
  • Kubernetes（K8s）是容器编排的事实标准——自动调度、健康监控、弹性伸缩、服务发现
  • Slurm在传统 HPC 和学术界仍有大量部署——队列模型天然适合训练作业
  • Gang Scheduling（群调度）是 AI 训练的关键需求——GPU 必须同时到齐，一个不能少
  • MIG（多实例 GPU）在硬件层面切分 GPU，让一张大卡可以被多个任务共享，提高利用率
  • NVIDIA 收购 Run:ai + GPU Operator标志着 GPU 资源调度正走向全栈智能化和平台化
