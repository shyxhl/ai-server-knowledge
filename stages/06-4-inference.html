阶段 6.4 · 推理引擎

场景：模型训好了，然后呢？

模型训完以后，真正的产品才开始——用户发来一个问题，模型要在几百毫秒内生成回复。每次这样的一次"回复"被称为一次推理（Inference，模型根据输入产生输出的过程）。

训练是"造车"，推理是"开车"。训练需要巨大的计算量但不需要实时响应——没人等着训练结果，跑几十天也无所谓。推理完全不同：用户每打一个字，模型就要实时响应；高峰期可能有上万个用户同时提问，系统一个都不能卡。两者的优化方向完全不同。

为什么推理也需要专门的引擎？直接用 PyTorch 跑模型不行吗？

行是行，但慢。PyTorch 是为"灵活性"设计的——研究者可以随便改模型结构、加新算子、做各种实验。但到了生产环境，模型是固定的，你不需要灵活性，你需要的是速度。TensorRT-LLM（NVIDIA 的推理优化引擎）可以把推理速度提升 5-10 倍——这意味着同样的硬件可以服务 5-10 倍的用户，或者同样的用户量只需要 1/5 的硬件。这笔账算下来，省下的 GPU 采购成本是千万级的。

关键认知：推理不是训练的"附属品"。对绝大多数 AI 公司来说，训练是成本中心（花钱训练模型），推理是利润中心（用户每次调用都产生收入或价值）。推理效率直接决定了商业模式能不能跑通——推理成本降一半，毛利率可能翻倍。

核心概念

KV Cache：推理的命脉

KV Cache（Key-Value Cache，键值缓存）是理解推理性能的第一把钥匙。要理解它为什么重要，先看大模型是怎么逐字生成文本的。

大模型生成文本是"一个字一个字蹦出来的"——这叫做自回归生成（Autoregressive Generation）。具体过程：

  1. 你输入"今天天气"，模型算一次，输出"真"
  2. 模型把完整的"今天天气真"再算一次，输出"好"
  3. 模型把"今天天气真好"再算一次，输出下一个字……

每生成一个字，模型都要"重新阅读"之前所有的字。如果每次重新读都要完整重算，那生成第 1000 个字需要重复计算前 999 个字的信息——慢得不像话。

KV Cache 就是用来解决这个问题的：模型在生成每个字的时候，会把中间计算结果（Key 和 Value 矩阵）缓存起来。生成下一个字时，老的 Token（文本的最小单位，可以理解为"字"或"词"的数字化表示）直接从缓存读，只需要为新 Token 计算一次。这样，生成第 1000 个字和第 1 个字的计算量几乎是相同的。

生活类比：
没有 KV Cache = 每次写一个字就要把整篇文章从头读一遍。
有 KV Cache = 手里拿着之前的笔记，写新字时翻一下笔记就行，不需要重读全文。

KV Cache 的问题是吃显存。一个 70B 参数的大模型（如 Llama 3 70B），处理 4096 个 Token 的上下文，KV Cache 可能占用约 2-4 GB 显存。对话越长，上下文越多，KV Cache 越大。一个长对话可以轻松吃掉所有可用 GPU 显存。这直接引出了推理领域需要解决的核心矛盾：速度、显存、吞吐量（单位时间内能处理多少请求）三者之间的平衡。

Continuous Batching：让 GPU 永远不闲着

在推理服务中，用户的请求是随机到达的。传统做法叫做静态批处理（Static Batching）：等服务积累一批请求，凑成一个 Batch（一批同时处理的请求），一起喂给 GPU，全部算完后再等下一批。

问题出在"等"上。这一批里的请求长短不一——短请求（比如回答"是"或"否"）早就算完了，但它不能离开，必须等这一批里最长的那个请求也完成，才能一起结束。GPU 在这段时间里是空转的。

Continuous Batching（连续批处理）的做法完全不同：不设边界。一旦某个请求生成完毕，立刻把它踢出 Batch，同时塞一个新请求进来。GPU 的 Batch 大小是动态变化的，永远保持满负荷。

生活类比：
静态批处理 = 餐厅等凑齐一桌子客人再开始做菜，有人先吃完也得等最慢的那个人吃完才能翻台。
连续批处理 = 有人吃完就走，立刻安排新客人坐下。座位永远坐满，翻台率最高。

这个方法是 vLLM（UC Berkeley 开源的推理引擎）的核心创新。相比静态批处理，连续批处理可以把吞吐量提升 2-10 倍。今天几乎所有主流推理引擎——TensorRT-LLM、SGLang、TGI——都实现了类似机制。

量化：把模型"压缩"

量化（Quantization）是指把模型参数从高精度数值格式转换成低精度格式，用极小的精度损失换取巨大的速度和内存收益。

训练大模型时，参数通常用 FP16（16 位浮点数，每个数占 2 个字节，能表示小数点后多位数字的数值格式）或 BF16 存储。推理时，可以把这些参数压到 INT8（8 位整数）/ INT4（4 位整数，每个数只占半个字节）：

精度每参数占用70B 模型显存需求相对速度
FP16（原始）2 字节约 140 GB基准 (1x)
INT81 字节约 70 GB约 1.5-2x
INT40.5 字节约 35 GB约 2-4x
生活类比：
把一张 RAW 格式照片（100MB，无损）压缩成 JPEG（5MB，有损但肉眼几乎看不出区别）。量化就是这个逻辑——模型"瘦身"到 1/4 的体积，回答质量几乎不变。

主流量化算法：

GPTQ

基于校准数据的逐层量化方法。用一批样本数据"观察"每一层的数据分布，按分布情况决定量化策略。需要在校准数据集上跑一次，量化完成后得到新的模型文件，直接加载使用。精度保持好，是生产环境的主流选择。

AWQ（激活感知权重量化）

发现了一个关键现象：模型中只有约 1% 的权重（参数）对结果影响巨大，直接量化这些"重要权重"会严重损失精度。AWQ 的做法是：先找出这些关键权重，在量化前把它们略微放大，量化后再还原——保护它们不受量化误差的伤害。效果通常比 GPTQ 还好。

INT4 量化后，一个 70B 参数的大模型从需要 4 张 H100（每张 80GB）变成只需要 2 张，甚至一张 H200（141GB）就能跑。成本直接减半。

技术要点

延迟与吞吐量的权衡

推理系统有两个互相矛盾的指标：

  • 延迟（Latency，从收到请求到第一个字输出的时间+后续每个字的生成间隔）：越低越好，用户体验的关键。人盯着屏幕等回复，超过 200 毫秒就会感觉"卡"。
  • 吞吐量（Throughput，单位时间内处理的总 Token 数）：越高越好，经济指标。吞吐量高意味着同样硬件能服务更多用户，单 Token 成本更低。

两者是矛盾的：Batch 越大，吞吐量越高（GPU 并行处理更多请求），但每个请求的延迟也会增加（排队等计算资源）。反之，Batch 越小，延迟越低但吞吐量也低。

这就是推理引擎的"甜点"问题——要根据业务场景找到平衡点：

  • 实时对话（ChatGPT 类产品）：优先延迟，用户不能等。Batch 保持较小，接受较低的吞吐量。
  • 离线批量处理（给大量文档做摘要、数据分类）：优先吞吐量。延迟不重要，没人等着看结果。Batch 拉满。
  • 代码补全（Copilot 类产品）：两个指标都敏感——用户敲代码时延迟不能超过 100ms，但请求量巨大。这是最难的场景。

推测解码：用小模型"猜"大模型

推测解码（Speculative Decoding）是一个巧妙的想法：大模型生成一个字很慢，因为每次都要跑完整个网络。但如果有一个小模型（参数量是大模型的 1/10 甚至 1/100）先用极快的速度"猜"出 5-10 个候选字，然后大模型一次性验证这 5-10 个字对不对，接受正确的、扔掉错误的——这就相当于大模型一次推理生成了多个字。

生活类比：
小模型是"草稿秘书"——快速拟一份草稿给老板。
大模型是"老板"——一眼扫过去，对的直接签字，错的划掉。秘书写了 10 句只错了 2 句，老板比亲自写快得多。

实际效果：在文本类任务（客服对话、文章续写）上可以获得 2-3 倍加速，且输出质量和大模型单独生成完全一致——因为最终是由大模型把关的。目前 TensorRT-LLM 和 vLLM 都内置了对推测解码的支持。

推理场景的 GPU 怎么选？

和训练集群完全不同。训练需要 NVLink、NVSwitch、InfiniBand——推理往往不需要：

NVIDIA L40S

48GB 显存，专为推理和视觉场景设计。价格远低于 H100，但 INT4/INT8 推理性能不差。适合中小模型（7B-70B）、批处理场景、图像生成推理。性价比之选。

NVIDIA H100

80GB 显存，训练和推理都能打。FP8 推理性能强劲，支持 Transformer Engine 加速。适合大模型（70B+）、对延迟敏感的实时对话场景。

NVIDIA H200

141GB 显存（HBM3e），带宽 4.8 TB/s。最大的卖点是超大的 KV Cache 容量。一个 H200 可以装下 H100 两倍的对话上下文。适合超长上下文推理、单机部署超大模型。

关键判断逻辑：

  • 显存带宽比算力更关键。推理的瓶颈通常不是计算速度（FLOPS），而是从显存读取 KV Cache 的速度。HBM3e 的高带宽在推理中是最有价值的规格。
  • 推理不需要高端网络。大部分推理是单 GPU 或少数 GPU 并行，GPU 之间通信极少。用普通以太网甚至不需要 RDMA，InfiniBand 纯属浪费。
  • 推理需要更大的显存。KV Cache 吃显存、Batch 大小吃显存、模型参数吃显存。显存不够就只能缩小 Batch 或量化，直接影响吞吐量和质量。

品牌视角：推理是 AI 的变现环节

从商业角度看，训练和推理是完全不同的账：

  • 训练是成本中心。花几千万甚至上亿买 GPU 集群，烧几个月电费，把模型训出来——这个过程中没有任何收入，纯支出。训完后 GPU 如果没有推理任务，可以直接关掉省电。
  • 推理是利润中心。用户每次调用 API 都产生收入或产品价值。推理的收入是持续的、累积的——一个模型训一次，可以被调用几十亿次。

这个区别直接反映在硬件采购策略上：

训练集群和推理集群的配置差异：

训练集群需要：高端网络（InfiniBand）、NVLink + NVSwitch（多 GPU 紧密耦合）、大量 GPU 同构互联、高带宽并行文件存储。

推理集群需要：大显存（容纳 KV Cache）、高显存带宽（不是高计算能力）、普通网络即可（以太网，不需要 RDMA）、单 GPU 或少数 GPU 即可。GPU 之间不需要高速互联，节点之间也不需要。存储要求也低得多。

这意味着，训练集群的硬件不适合直接拿来做推理——花大价钱买了 InfiniBand 和 NVSwitch，推理时完全用不上。正确的做法是配两套集群，或者在同一个集群中划分训练区和推理区，各自用不同的网络和 GPU 配置。

从 GPU 选型角度看：中国市场受制裁影响，H100/H200 不容易拿到。但推理场景有更多替代方案：

  • L40S 不受制裁限制，是推理的理想选择。48GB 显存足够跑 INT4 量化后的 70B 模型。
  • 国产 GPU（华为昇腾 910B、寒武纪 MLU370）在推理场景下差距较小——推理不需要复杂的分布式通信，对软件生态的依赖也比训练低。国产替代在推理领域比训练领域现实得多。
  • 甚至消费级 GPU（如 RTX 4090、RTX 5090）在某些小规模推理场景下也能用——24GB 到 32GB 显存，INT4 量化后跑 7B-13B 模型绰绰有余。成本只有 H100 的 1/10 到 1/20。

一张图总结

推理引擎核心要点

推理是 AI 的"最后一公里"——模型训得再好，推理慢了用户不会用。

  • KV Cache 是推理的命脉。通过缓存中间结果，让长文本生成速度不再随长度线性恶化。代价是吃显存——长对话会吃掉大量 GPU 显存。
  • 连续批处理是效率倍增器。动态管理请求队列，GPU 永远保持满载。vLLM 凭这项创新改变了整个推理领域的工程实践。
  • 量化让大模型能在手机上跑。INT4 量化可将模型体积压缩到 1/4，精度损失不到 1%。FP16 是"原图"，INT4 是高质量 JPEG。
  • TensorRT-LLM 追求极致性能，vLLM 追求开源易用。两者经常互补——vLLM 做原型，TensorRT-LLM 做生产。
  • 训练是成本中心，推理是利润中心。训练集群和推理集群的硬件配置完全不同——推理不需要 InfiniBand，不需要 NVSwitch，但需要更大的显存。

  • TensorRT-LLM 官方文档 — NVIDIA 的推理引擎完整指南。涵盖了图编译优化、In-flight Batching、推测解码等高级特性。技术深度足够，但需要一定基础。
  • AWQ: Activation-aware Weight Quantization for On-Device LLM Compression and Acceleration — 当前最主流的量化方法之一的技术论文。讲清楚了为什么保护 1% 的关键参数比均匀量化效果好得多。
